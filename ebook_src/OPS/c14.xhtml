THE HELL OF LIBRARIES
图书馆的地狱

Libraries make it possible for you to reuse code in multiple applications. With Windows, libraries have a long history, and architecture guidelines have taken different directions with newer technologies. Before .NET, dynamic link libraries (DLLs) could be shared between different applications. These DLLs have been installed in a shared directory. It wasn't possible to have multiple versions of these libraries on the same system, but they should have been upward compatible. Of course, this wasn't always the case. In addition, there were problems with application installations that did not pay attention to the guidelines and replaced a shared library with an older one. This was known as DLL hell.
图书馆使您能够在多个应用程序中重用代码。在 Windows 中，图书馆有着悠久的历史，并且随着新技术的出现，架构指南也采取了不同的方向。在.NET 之前，动态链接库（DLL）可以在不同的应用程序之间共享。这些 DLL 被安装在一个共享目录中。在同一系统中不可能有这些库的多个版本，但它们应该是向上兼容的。当然，这并不总是如此。此外，还有应用程序安装时没有遵循指南，并用旧版本替换了共享库的问题。这被称为 DLL 地狱。

.NET tried to solve this with assemblies. Assemblies are libraries that could be shared. In addition to normal DLLs, assemblies contain extensible metadata with information about the library and a version number, and it's possible to install multiple versions side by side in the global assembly cache. Microsoft tried to fix versioning issues, but this added another layer of complexity.
.NET 试图通过程序集来解决这些问题。程序集是可以共享的库。除了普通的 DLL 之外，程序集还包含可扩展的元数据，其中包含有关库的信息和版本号，并且可以在全局程序集缓存中并排安装多个版本。微软试图解决版本问题，但这增加了另一层的复杂性。

Let's assume you're using libraries A and B from your application X (see Figure 14-1). Application X references version 1.1 from library A and version 1.0 from library B. The issue is that library B references library A as well, but it references a different version—version 1.0. One process can have only one version of a library loaded. What version of the library A is loaded into the process? In this case, library B is used before library A, so version 1.0 wins. This is a big issue as soon as application X needs to use library A itself.
让我们假设你的应用程序 X 使用了库 A 和库 B（见图 14-1）。应用程序 X 引用了库 A 的 1.1 版本和库 B 的 1.0 版本。问题是库 B 也引用了库 A，但它引用的是不同版本——1.0 版本。一个进程只能加载一个版本的库。那么，库 A 的哪个版本被加载到进程中？在这种情况下，因为库 B 在库 A 之前使用，所以 1.0 版本获胜。当应用程序 X 需要使用库 A 时，这是一个大问题。

To avoid this issue, you could configure assembly redirects. You can define an assembly redirect for application X to load version 1.1 from library A. Library B then needs to use version 1.1 from library A as well. As long as library A is upward-compatible, this shouldn't be an issue.
为了避免这个问题，你可以配置程序集重定向。你可以为应用程序 X 定义一个程序集重定向，使其加载库 A 的 1.1 版本。然后库 B 也需要使用库 A 的 1.1 版本。只要库 A 是向上兼容的，这不应该是个问题。

Of course, compatibility doesn't always exist, and issues can be more complex. Publishers of components can create a publisher policy to define redirects with a policy that is delivered with the library itself. This redirect can be overridden by the application. There's a lot of complexity with this, which resulted in assembly hell.
当然，兼容性并不总是存在，问题可能更复杂。组件的发布者可以创建一个发布者策略，通过随库本身一起交付的策略来定义重定向。这个重定向可以被应用程序覆盖。这其中有很多复杂性，导致了程序集地狱。

NOTE With the new .NET, there's no global sharing of assemblies as it was with the .NET Framework. The Global Assembly Cache (GAC) is no longer used.
注意：在新的.NET 中，不再像.NET Framework 那样进行全局共享程序集。全局程序集缓存（GAC）不再使用。

NuGet packages add another abstraction layer to libraries. A NuGet package can contain multiple versions of one or more assemblies, along with other stuff, such as automatic configuration of assembly redirects.
NuGet 包为库添加了另一层抽象。一个 NuGet 包可以包含一个或多个程序集的多个版本，以及其他内容，例如程序集重定向的自动配置。

Instead of waiting for new .NET Framework releases, you could add functionality via NuGet packages, which allowed for faster updates of the packages. NuGet packages are a great delivery vehicle. Some libraries, such as Entity Framework, switched to NuGet to allow for faster updates than the .NET Framework offered.
与其等待新的.NET Framework 版本发布，你可以通过 NuGet 包添加功能，这允许更快地更新包。NuGet 包是一个极好的交付工具。一些库，如 Entity Framework，切换到 NuGet 以提供比.NET Framework 更快的更新。

However, there are some issues with NuGet. Sometimes, a failure occurs when you add NuGet packages to projects. NuGet packages might not be compatible with the project. When adding packages is successful, sometimes the package makes some incorrect configuration with the project—for example, wrong binding redirects. This results in the feeling of NuGet package hell. The problems from DLLs moved to different abstraction layers and are indeed different. With newer NuGet versions and advancements in NuGet, Microsoft has tried to solve the issues with NuGet—and succeeded in many aspects.
然而，NuGet 存在一些问题。有时在将 NuGet 包添加到项目中时会发生失败。NuGet 包可能与项目不兼容。即使添加包成功，有时包会与项目进行一些错误的配置——例如错误的绑定重定向。这导致了 NuGet 包地狱的感觉。从 DLL 转移到不同的抽象层的问题确实是不同的。随着 NuGet 版本的更新和 NuGet 的进步，微软已经尝试解决 NuGet 的问题——并在许多方面取得了成功。

Directions in the architecture of .NET Core also changed. With .NET Core, packages have been made more granular. For example, with the .NET Framework, the Console class is inside the mscorlib assembly, which is an assembly needed by every .NET Framework application. Of course, not every .NET application needs the Console class. With .NET Core, a separate package System.Console exists that contains the Console class and a few related classes. The goal was to make it easier to update and select what packages are really needed. With some beta versions of .NET Core 1.0, the project files contained a large list of packages, which didn't make development easier. Just before the release of .NET Core 1.0, Microsoft introduced meta-packages (or reference-packages). A meta-package doesn't include code; it includes a list of other packages. A target framework moniker such as net5.0 defines a list of packages and APIs that are readily available for the application without the need to add NuGet packages.
.NET Core 的架构方向也发生了变化。使用.NET Core 时，包被做得更加细粒度。例如，在.NET Framework 中， Console 类位于 mscorlib 程序集中，这是一个每个.NET Framework 应用程序都需要的程序集。当然，并非每个.NET 应用程序都需要 Console 类。在.NET Core 中，存在一个单独的包 System.Console ，其中包含 Console 类和几个相关的类。其目标是为了更容易地更新和选择真正需要的包。在.NET Core 1.0 的一些测试版本中，项目文件包含了一个大量的包列表，这并没有使开发更容易。在.NET Core 1.0 发布之前，微软引入了元包（或引用包）。一个元包不包含代码，它包含其他包的列表。一个目标框架明器（如 net5.0）定义了一组应用程序可以立即使用的包和 API，而无需添加 NuGet 包。

This chapter goes into the detail of assemblies and NuGet packages, explains how to share code using .NET Standard libraries, and also explains differences with Windows Runtime components.
本章详细介绍了程序集和 NuGet 包，解释了如何使用.NET Standard 库共享代码，还解释了与 Windows Runtime 组件的差异。

ASSEMBLIES    程序集

An assembly is a library or executable that includes additional metadata. Using the new .NET, the application containing the Main method is created as a library with the file extension .dll. This DLL needs a hosting process to load this library, which you accomplish by using dotnet run, or just dotnet from a production environment. When you create stand-alone applications with .NET, different executables are created for every platform to load the library.
一个程序集是一个包含额外元数据的库或可执行文件。使用新的.NET，包含 Main 方法的程序被创建为一个具有 .dll 文件扩展名的库。这个 DLL 需要一个宿主进程来加载这个库，你可以通过使用 dotnet run ，或者从生产环境中直接使用 dotnet 来实现。当你使用.NET 创建独立应用程序时，会为每个平台创建不同的可执行文件来加载库。

Let's take a look at a simple “Hello, World!” console application created in the directory ConsoleApp, using this command:
让我们看一下在 ConsoleApp 目录中创建的简单的“Hello, World!”控制台应用程序，使用以下命令： 

> dotnet new console -o ConsoleApp

After building the application, the DLL can be found in the bin/debug/net5.0 directory. The net5.0 directory depends on the target framework listed in the csproj project file.
构建应用程序后，DLL 可以在 bin/debug/net5.0 目录中找到。 net5.0 目录取决于 csproj 项目文件中列出的目标框架。

NOTE You usually start the application using the dotnet bootstrapper (dotnet ConsoleApp.dll). On a Windows system, you'll also find the file ConsoleApp.exe. This is a Windows-specific file that can be used to start the application as well. It's just a bootstrapper to load the binary. On a Linux system, you'll find a similar bootstrapper that's Linux specific without the exe file extension.
注意 你通常使用 dotnet 引导程序（ dotnet ConsoleApp.dll ）启动应用程序。在 Windows 系统上，你还会找到文件 ConsoleApp.exe 。这是一个特定于 Windows 的文件，也可以用来启动应用程序。它只是一个加载二进制的引导程序。在 Linux 系统上，你会找到一个类似的引导程序，它是特定于 Linux 的，并且没有 exe 文件扩展名。

To read assemblies, you can install the .NET tool IL Disassembler (ildasm) using this:
要读取程序集，你可以使用以下方式安装 .NET 工具 IL Disassembler (ildasm)： 

> dotnet tool install dotnet-ildasm -g

After installing this tool, it can show metadata and IL code of assemblies. For easier reading of the result, the output is written to a text file with the -o option.
安装该工具后，它可以显示程序集的元数据和 IL 代码。为了更方便地阅读结果，输出会通过 -o 选项写入文本文件。 

> dotnet ildasm ConsoleApp.dll -o output.txt

In the output, you'll find the section .assembly "ConsoleApp" that shows attribute values for the attributes AssemblyCompany, AssemblyConfiguration, AssemblyFileVersion, AssemblyInformationalVersion, AssemblyProduct, and AssemblyTitle.
在输出中，你会找到显示 AssemblyCompany 、 AssemblyConfiguration 、 AssemblyFileVersion 、 AssemblyInformationalVersion 、 AssemblyProduct 和 AssemblyTitle 属性的值的部分 .assembly "ConsoleApp" 。

NOTE In case you also have the .NET Framework SDK installed, you can find an older version of the ildasm tool with a graphical output. You can still use the old version of this tool with newer .NET libraries because the metadata information is still the same between .NET Framework and .NET libraries.
注意 如果你同时安装了 .NET Framework SDK，你可以找到一个带有图形输出的旧版本 ildasm 工具。你仍然可以使用这个旧版本工具来处理较新的 .NET 库，因为 .NET Framework 和 .NET 库之间的元数据信息是相同的。

CREATING AND USING LIBRARIES
创建和使用库

To use the same code with multiple projects, you create libraries. Using the .NET CLI, you can create generic class libraries, class libraries for WPF, Windows Forms, and Razor. Razor class libraries are covered in Chapter 26, “Razor Pages and MVC.” When you create a class library with dotnet new classlib, by default, a library is created for .NET 5 and later versions, but you can also choose to select a different target framework. To make the correct choice, you have to ask, “With which application type and version of the framework should the library be shared?” There's another question you should ask: “Which APIs do I want to use in the library?” To answer both questions, it helps to understand parts of the history of .NET.
要使用同一代码于多个项目，你创建库。使用 .NET CLI，你可以创建通用类库、WPF 类库、Windows Forms 类库和 Razor 类库。Razor 类库在“第 26 章，Razor 页面和 MVC”中介绍。当你使用 dotnet new classlib 创建类库时，默认情况下会创建适用于 .NET 5 及更高版本的库，但你也可以选择选择不同的目标框架。要做出正确的选择，你必须问自己，“这个库应该与哪种应用程序类型和框架版本共享？” 你还应该问另一个问题：“我想在库中使用哪些 API？” 要回答这两个问题，了解 .NET 的一些历史背景会有所帮助。

If you created a .NET Framework class library, this library could be used only with .NET Framework applications. When Silverlight (.NET applications running in a browser with the help of an add-in) came along, it was interesting to share code between Silverlight and WPF applications. Silverlight (code-named WPF-E, WPF Everywhere) offered limited functionality compared to the full .NET Framework. Microsoft defined the Portable Class Library to share code with these technologies. Later, Xamarin allowed creating mobile applications for Android and iOS and used this library type as well. Depending on the platform and version selection, different APIs are available. The more platforms and the older the version chosen, the fewer APIs are available. As more and more platforms have been added, this increased the complexity of the definitions and also increased complexity on using portable libraries from portable libraries.
如果你创建了一个.NET Framework 类库，这个库只能用于.NET Framework 应用程序。当 Silverlight（在浏览器中通过插件运行的.NET 应用程序）出现时，在 Silverlight 和 WPF 应用程序之间共享代码变得很有趣。与完整的.NET Framework 相比，Silverlight（代码名为 WPF-E，WPF Everywhere）的功能有限。微软定义了便携类库来与这些技术共享代码。后来，Xamarin 允许为 Android 和 iOS 创建移动应用程序，并也使用了这种库类型。根据平台和版本选择的不同，可用的 API 也不同。平台越多、选择的版本越老，可用的 API 就越少。随着越来越多的平台被添加，这增加了定义的复杂性，同时也增加了从便携类库使用便携类库的复杂性。

.NET Standard provides a replacement for portable libraries. Instead of APIs defined in a matrix, .NET Standard has a simpler definition of the APIs available. With every version of the .NET Standard, additional APIs are added. APIs are never removed from the .NET Standard.
.NET Standard 提供了对可移植库的替代方案。它不再使用矩阵定义 API，而是以更简单的形式定义可用的 API。随着每个 .NET Standard 版本的发布，会添加新的 API。API 永远不会从 .NET Standard 中移除。

Starting with .NET 5.0, the .NET Standard is not further developed. From now on, you can create .NET 5.0 libraries, which can be used from .NET 5 upward. So, a .NET 6 and .NET 7 application can use the .NET 5 library. In case you create only new applications, you can skip the next section on .NET Standard. However, you probably still have to support or extend .NET Framework, UWP, Xamarin, and other application types probably for many years to come, in which case the .NET Standard is still relevant and important.
从 .NET 5.0 开始，.NET Standard 不再进一步开发。从现在起，你可以创建可在 .NET 5 及更高版本中使用的 .NET 5.0 库。因此，.NET 6 和 .NET 7 应用程序可以使用 .NET 5 库。如果你只创建新应用程序，可以跳过下一节关于 .NET Standard 的内容。然而，你可能仍需在未来多年内支持或扩展 .NET Framework、UWP、Xamarin 等其他应用程序类型，在这种情况下，.NET Standard 仍然具有相关性和重要性。

.NET Standard

The .NET Standard makes a linear definition of APIs available, which is different than the matrix definition with APIs that was available for portable libraries. Every version of the .NET Standard adds APIs, and APIs are never removed.
.NET Standard 提供了 API 的线性定义，这与可移植库中使用的矩阵定义 API 不同。每个 .NET Standard 版本都会添加 API，且 API 永远不会被移除。

The higher the version of the .NET Standard, the more APIs you can use. However, the .NET Standard doesn't implement the APIs; it just defines the APIs that need to be implemented by a .NET platform. This can be compared to interfaces and concrete classes. An interface just defines a contract for members that need to be implemented by a class. Similarly, the .NET Standard specifies what APIs need to be available, and a .NET platform—supporting a specific version of the standard—needs to implement these APIs.
版本越高，.NET Standard 提供的 API 就越多。然而，.NET Standard 本身并不实现这些 API，它只是定义了.NET 平台需要实现的 API。这可以与接口和具体类进行类比。接口只是定义了类需要实现的成员的契约。类似地，.NET Standard 指定了需要提供的 API，而支持特定版本标准的.NET 平台需要实现这些 API。

You can find which APIs are available for each standard version, as well as the differences between the standards, at https://github.com/dotnet/standard/tree/master/docs/versions.
您可以在 https://github.com/dotnet/standard/tree/master/docs/versions 处找到每个标准版本提供的 API 以及标准之间的差异。

Every version of the .NET Standard adds APIs to the standard:
每个.NET Standard 版本都会向标准中添加 API：

  • .NET Standard 1.1 added 2,414 APIs to .NET Standard 1.0.
    .NET Standard 1.1 向 .NET Standard 1.0 添加了 2,414 个 API。
  • Version 1.2 added just 46 APIs.
    版本 1.2 仅增加了 46 个 API。
  • Version 1.3 added 3,314 APIs.
    版本 1.3 添加了 3,314 个 API。
  • Version 1.4 added only 18 Cryptography APIs.
    版本 1.4 仅添加了 18 个加密 API。
  • Version 1.5 mainly enhanced reflection support and added 242 APIs.
    版本 1.5 主要增强了反射支持，并添加了 242 个 API。
  • Version 1.6 added more Cryptography APIs and enhanced regular expressions, with a total of 146 additional APIs.
    版本 1.6 添加了更多加密 API 并增强了正则表达式，总共添加了 146 个 API。
  • Version 2.0 added 19,507 APIs.
    版本 2.0 添加了 19,507 个 API。
  • Version 2.1 added enhancements requiring to update the runtime—for example, with support for default interface methods.
    版本 2.1 增加了需要更新运行时的改进——例如，支持默认接口方法。

With .NET Standard 2.0, Microsoft made a big investment to make it easier to move legacy applications to .NET Core: 19,507 APIs have been added. Many of these APIs were not new. Some were already implemented with .NET Framework 4.6.1. For example, old APIs like DataSet, DataTable, and others have been available with the .NET Standard since version 2.0. This was a move to make it easier to bring legacy applications to the new .NET. A huge investment was needed for .NET Core, because .NET Core 2.0 implements the .NET Standard 2.0.
在 .NET Standard 2.0 中，微软进行了重大投资，以简化将遗留应用程序迁移到 .NET Core 的工作：添加了 19,507 个 API。其中许多 API 并非全新。一些 API 已经在 .NET Framework 4.6.1 中实现。例如，像 DataSet 、 DataTable 等旧 API 自 2.0 版本起就已在 .NET Standard 中可用。这是为了让遗留应用程序更容易迁移到新的 .NET。为 .NET Core 所需的巨大投资，是因为 .NET Core 2.0 实现了 .NET Standard 2.0。

What APIs are not in the standard? Platform-specific APIs are not part of .NET Standard. For example, Windows Presentation Foundation (WPF) and Windows Forms define Windows-specific APIs that will not make it into the standard. You can, however, create WPF and Windows Forms applications and use .NET Standard libraries from there. You cannot create .NET Standard libraries that contain WPF or Windows Forms controls.
哪些 API 不在标准中？平台特定的 API 不属于 .NET Standard。例如，Windows Presentation Foundation (WPF) 和 Windows Forms 定义了 Windows 特定的 API，这些 API 不会包含在标准中。然而，您可以从 WPF 和 Windows Forms 应用程序中创建应用程序并使用 .NET Standard 库。您无法创建包含 WPF 或 Windows Forms 控件的 .NET Standard 库。

Let's discuss the platform support of .NET Standard. The Microsoft documentation at https://docs.microsoft.com/en-us/dotnet/standard/net-standard lists the details on what .NET Standard version supports which platform version. If you need to support .NET Framework 4.7.2 or up (including a check for the footnote on issues with .NET Framework 4.6.1), you can use .NET Standard 2.0 but not 2.1. With .NET Framework 4.6, you're restricted to .NET Standard 1.3. Using Windows 10 starting with version 10.0.16299, you can use .NET Standard 2.0. With the Mono platform, version 6.4 supports .NET Standard 2.1. .NET Standard 2.1 is also supported from .NET Core 3.0 and up.
我们来讨论一下 .NET Standard 的平台支持情况。微软文档 https://docs.microsoft.com/en-us/dotnet/standard/net-standard 列出了 .NET Standard 版本支持哪些平台版本。如果你需要支持 .NET Framework 4.7.2 或更高版本（包括检查有关 .NET Framework 4.6.1 问题的脚注），你可以使用 .NET Standard 2.0，但不能使用 2.1。对于 .NET Framework 4.6，你只能使用 .NET Standard 1.3。从 Windows 10 版本 10.0.16299 开始，你可以使用 .NET Standard 2.0。在 Mono 平台上，版本 6.4 支持 .NET Standard 2.1。从 .NET Core 3.0 开始，也支持 .NET Standard 2.1。

NOTE To support the most platforms possible, you need to select a lower .NET Standard version. To have more APIs available, select a higher .NET Standard version.
注意 为了支持尽可能多的平台，您需要选择较低的 .NET Standard 版本。为了获得更多的 API，选择较高的 .NET Standard 版本。

Creating a .NET Library
创建 .NET 库

To create a .NET library, you can use the .NET Core CLI tools with the following command:
要创建 .NET 库，你可以使用以下命令和 .NET Core CLI 工具： 

> dotnet new classlib -o SampleLib

默认情况下，如果你安装了 .NET 5，它会创建一个 .NET 5 库。你可以提供选项 --framework 并添加 netstandard2.1 或 netstandard2.0 来创建具有特定版本的 .NET Standard 库。你可以在项目文件中稍后更改版本号。

<Project Sdk="Microsoft.NET.Sdk">

You can change the version of the target framework of the library by changing the value of the TargetFramework element. Later in this chapter, this configuration is enhanced to support multiple frameworks with one library.
你可以通过更改 TargetFramework 元素的值来更改库的目标框架版本。在本章后面，此配置将得到增强，以使用一个库支持多个框架。

NuGet Sources    NuGet 源

Where are the packages coming from? www.nuget.org is a public server where Microsoft and third parties upload .NET packages. After the packages have been downloaded from the NuGet server for the first time, the packages are stored in the user profile. Thus, it becomes a lot faster to create another project with the same packages.
这些包是从哪里来的？ www.nuget.org 是一个公共服务器，Microsoft 和第三方上传 .NET 包的地方。当包第一次从 NuGet 服务器下载后，这些包会存储在用户配置文件中。因此，使用相同的包创建另一个项目会快得多。

On Windows, the directory for the packages in the user profile is %userprofile%\.nuget\packages. Other temporary directories are used as well. To get all the information about these directories, it's best to install the NuGet command-line utility, which you can download from https://dist.nuget.org/.
在 Windows 系统中，用户配置文件中的包目录是 %userprofile%\.nuget\packages 。同时也会使用其他临时目录。若要获取这些目录的所有信息，最好安装 NuGet 命令行工具，该工具可从 https://dist.nuget.org/ 下载。

To see the folders for the global packages, the HTTP cache, and the temp packages, you can use nuget locals :
要查看全局包、HTTP 缓存和临时包的文件夹，可以使用 nuget locals ： 

> nuget locals all -list

In some companies, it's permissible to use only packages that have been approved and are stored in a local NuGet server. The default configuration for the NuGet server is in the file NuGet.Config in the directory %appdata%/nuget.
在某些公司，只允许使用已批准并存储在本地 NuGet 服务器中的包。NuGet 服务器的默认配置位于 %appdata%/nuget 目录下的 NuGet.Config 文件中。

A default configuration looks similar to the following NuGet.Config file. Packages are loaded from https://api.nuget.org.
默认配置类似于以下 NuGet.Config 文件。包从 https://api.nuget.org 加载。 


<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <packageSources>
    <add key="nuget.org" value="https://api.nuget.org/v3/index.json" 
         protocolVersion="3"/>
  </packageSources>
</configuration>

You can change the defaults by adding and removing package sources. Instead of changing the defaults, you can create a NuGet configuration file for the project:
你可以通过添加和删除包源来更改默认设置。与其更改默认设置，你可以为项目创建一个 NuGet 配置文件： 

> dotnet new nugetconfig

Microsoft doesn't store packages from the daily build on the main NuGet server. To use daily builds of .NET Core NuGet packages, you need to configure other NuGet servers. For example, to use .NET 6 daily feeds, you can add the .NET 6 feed. The following commands also add a feed to a local directory:
微软不会在主 NuGet 服务器上存储每日构建的包。要使用.NET Core NuGet 包的每日构建版本，你需要配置其他 NuGet 服务器。例如，要使用.NET 6 每日源，你可以添加.NET 6 源。以下命令也将一个源添加到本地目录： 

> dotnet nuget add source -n dotnet6   
  https://dnceng.pkgs.visualstudio.com/public/_packaging/dotnet6/nuget/v3/index.json
> dotnet nuget add source -n local c:\mypackages

The following NuGet file adds a local directory in addition to the public NuGet server and the .NET 6 feed:
以下 NuGet 文件除了公共 NuGet 服务器和.NET 6 源之外，还添加了一个本地目录： 

<?xml version="1.0" encoding="utf-8"?>
<configuration>
  <packageSources>
    <clear/>
    <add key="nuget" value="https://api.nuget.org/v3/index.json"/>
    <add key="local packages" value="C:\mypackages"/>
    <add key="dotnet6" value=
      "https://dnceng.pkgs.visualstudio.com/public/_packaging/dotnet6/nuget/v3/    index.json" 
   />
  </packageSources>
</configuration>

CREATING NUGET PACKAGES    创建 NuGet 包

After you created a library and an application where this library is referenced, it's time to create NuGet packages on your own. NuGet packages can be created easily by using the .NET Core CLI tools and Visual Studio.
创建好库和引用该库的应用后，就可以自行创建 NuGet 包了。使用.NET Core CLI 工具和 Visual Studio 可以轻松创建 NuGet 包。

NuGet Packages with the Command Line
使用命令行创建 NuGet 包

Metadata information about the NuGet package can be added to the project file csproj as shown earlier in the “Assemblies” section. To create a NuGet package from the command line, you can use the dotnet pack command (started from the project directory):
如“程序集”部分所示，可以将 NuGet 包的元数据信息添加到项目文件 csproj 中。要从命令行创建 NuGet 包，可以使用 dotnet pack 命令（从项目目录启动）： 

> dotnet pack --configuration Release

Remember to set the configuration. By default, the Debug configuration is built. After a successful packaging, you can find the NuGet package in the directory bin/Release or related directories, depending on the selected configuration with the file extension .nupkg. A .nupkg file is a zip file that contains the binary with additional metadata. You can rename this file to a zip file to see its contents.
请记得设置配置。默认情况下，构建的是 Debug 配置。成功打包后，你可以在 bin/Release 目录或相关目录中找到 NuGet 包，具体取决于所选配置，文件扩展名为 .nupkg 。一个 .nupkg 文件是一个包含二进制文件和额外元数据的 zip 文件。你可以将这个文件重命名为 zip 文件来查看其内容。

The package file includes the version number. The version number is taken from the Version value in the project file.
包文件包含版本号。版本号取自项目文件中的 Version 值。

You can copy the generated NuGet package to a folder on your system or to a network share to make it available to your team. This command copies the package to a subfolder with the name of the package:
您可以将生成的 NuGet 包复制到系统上的文件夹或网络共享中，以便您的团队使用。此命令将包复制到以包名称命名的子文件夹中： 

> nuget add bin\Release\SampleLib.5.0.1.nupkg -s c:\MyPackages

To use the folder c:\MyPackages, the NuGet.config file can be changed to include this package source as shown in the section “NuGet Sources.” You also can reference the folder directly by using the dotnet add package command:
要使用文件夹 c:\MyPackages ，可以将 NuGet.config 文件更改为包含此包源，如“NuGet 源”部分所示。您还可以通过使用 dotnet add package 命令直接引用该文件夹： 

> dotnet add package SampleLib --source c:/MyPackages

By default, the latest released version of the package is referenced from the project where you add the package. With the option --version, you can specify the exact version to add.
默认情况下，从您添加包的项目中引用最新发布的版本。使用选项 --version ，您可以指定要添加的确切版本。

To create prerelease packages, you just need to add a postfix to the version number—for example, 5.0.1−alpha, 5.0.1−alpha.2, 5.0.1−beta.1, 5.0.1−beta.2, 5.0.1−preview3, 5.0.1−rc1. Reverse alphabetical order is used to identify the newest version of the prerelease packages, so beta is newer than alpha, and rc is newer than preview. The same version number without a postfix is considered a release and thus a newer version. To add a prerelease package version to a project, you need to add the --prerelease option with dotnet add package.
要创建预发布包，您只需在版本号后添加后缀——例如， 5.0.1−alpha 、 5.0.1−alpha.2 、 5.0.1−beta.1 、 5.0.1−beta.2 、 5.0.1−preview3 、 5.0.1−rc1 。使用反向字母顺序来识别预发布包的最新版本，因此 beta 比第 alpha 新， rc 比第 preview 新。不带后缀的相同版本号被视为正式发布版本，因此是较新版本。要将预发布包版本添加到项目中，您需要添加带 --prerelease 选项的 dotnet add package 。

Supporting Multiple Platforms
支持多平台

.NET 5 includes a lot more packages and APIs compared to what's available with .NET Standard 2.0. In case you want to offer new features for .NET 5+ clients but still want to support clients with .NET Standard 2.0 support, you can create a NuGet package containing multiple binaries with different version support.
与 .NET Standard 2.0 相比，.NET 5 包含更多的包和 API。如果您想为 .NET 5+ 客户提供新功能，但仍想支持 .NET Standard 2.0 客户，可以创建一个包含多个二进制文件且支持不同版本的 NuGet 包。

To support multiple frameworks, in the project file, you can change the TargetFramework element to TargetFrameworks. All the target framework monikers for the target frameworks where binaries should be created are listed within.
要支持多个框架，在项目文件中，您可以将 TargetFramework 元素更改为 TargetFrameworks 。所有目标框架的框架标识符（target framework monikers）都列在其中，用于创建二进制文件的目标框架。

NOTE The list of target framework monikers is shown at https://docs.microsoft.com/dotnet/standard/frameworks.
注意 目标框架限定符列表显示在 https://docs.microsoft.com/dotnet/standard/frameworks 处。

The example adds the target framework monikers net5.0 and netstandard2.0. With conditional settings, constants are defined with the element DefineConstants. You can use these constants with preprocessor directives to create different code between the different frameworks. The C# version is specified with the LangVersion element. Without this, the default C# versions based on the different frameworks would be used. With .NET 5, it's C# 9; with .NET Standard 2.0, it's C# 7.3. Changing the C# version to 9.0 doesn't mean you can use all the features with .NET Standard 2.0. For example, default interface members are not possible and can be used only for writing conditional C# code using the defined constant DOTNET50 (project file CreateNuGet/SampleLib/SampleLib.csproj):
示例添加了目标框架限定符 net5.0 和 netstandard2.0 。通过条件设置，使用元素 DefineConstants 定义常量。您可以使用这些常量与预处理指令来在不同的框架之间创建不同的代码。C# 版本通过 LangVersion 元素指定。没有这个元素，将使用基于不同框架的默认 C# 版本。对于 .NET 5，它是 C# 9；对于 .NET Standard 2.0，它是 C# 7.3。将 C# 版本更改为 9.0 并不意味着您可以在 .NET Standard 2.0 中使用所有功能。例如，默认接口成员是不可能的，并且只能用于使用定义的常量 DOTNET50（项目文件 CreateNuGet/SampleLib/SampleLib.csproj ）编写条件 C# 代码： 

<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <TargetFrameworks>net5.0;netstandard2.0</TargetFrameworks>
    <Nullable>enable</Nullable>
    <LangVersion>9.0</LangVersion>
  </PropertyGroup>
 
  <PropertyGroup Condition="'$(TargetFramework)'=='netstandard2.0'">
    <DefineConstants>NETSTANDARD_20</DefineConstants>
  </PropertyGroup>
 
  <PropertyGroup Condition="'$(TargetFramework)'=='net5.0'">
    <DefineConstants>DOTNET50</DefineConstants>
  </PropertyGroup>
 
  <ItemGroup Condition="'$(TargetFramework)' == 'netstandard2.0'">
    <PackageReference Include="System.Text.Json" Version="5.0.1" />
  </ItemGroup>
 
</Project>

The project file also lists a conditional reference to the NuGet package System.Text.Json. This package is already referenced from the target framework moniker net5.0. It's not part of .NET Standard 2.0. Because this package also supports .NET Standard 2.0, it can be added to the project when building the library for .NET Standard 2.0. You can add conditional package references using the --framework option to specify the target framework moniker. This adds the package reference shown in the previous code snippet.
项目文件还列出了一个条件引用的 NuGet 包 System.Text.Json 。该包已经在目标框架标识符 net5.0 中被引用。它不是 .NET Standard 2.0 的一部分。因为这个包也支持 .NET Standard 2.0，所以在为 .NET Standard 2.0 构建库时可以将其添加到项目中。您可以使用 --framework 选项来指定目标框架标识符，从而添加条件包引用。这添加了先前代码片段中显示的包引用。 

> dotnet add package --framework netstandard2.0 System.Text.Json

With the C# code, preprocessor directives are used to decide between .NET Standard 2.0 and .NET 5.0 code. The Show method returns different values depending on how the code was built. The JsonSerializer defined in the System.Text.Json namespace can now be used with both .NET versions. It's already part of .NET 5, and with .NET Standard 2.0, the library has been added (code file CreateNuGet/SampleLib/Demo.cs):
在 C#代码中，使用预处理指令在 .NET Standard 2.0 和 .NET 5.0 代码之间进行选择。 Show 方法根据代码的构建方式返回不同的值。在 System.Text.Json 命名空间中定义的 JsonSerializer 现在可以与两个 .NET 版本一起使用。它已经是 .NET 5 的一部分，并且在使用 .NET Standard 2.0 时，库已被添加（代码文件 CreateNuGet/SampleLib/Demo.cs ）： 

using System.Text.Json;
 
namespace SampleLib
{
  public class Demo
  {
#if NETSTANDARD20
    private static string s_info = ".NET Standard 2.0";
#elif NET50
    private static string s_info = ".NET 5.0";
#else
    private static string s_info = "Unknown";
#endif
 
    public static string Show() => s_info;
 
    public string GetJson(Book book) =>
      JsonSerializer.Serialize(book);
  }
}

With this setup, you build the application with multiple target frameworks, and a DLL for every target framework is created. You can also build a library just for one of the specified target frameworks setting the --framework option. When you create a NuGet package, one package is created that contains all the libraries.
通过这种设置，您使用多个目标框架构建应用程序，并为每个目标框架创建一个 DLL。您也可以只为指定的一个目标框架构建库，通过设置 --framework 选项。当您创建 NuGet 包时，会创建一个包含所有库的包。

When creating the .NET console application, you can build the application for multiple target frameworks as well. Like the library before, with the console application, you configure multiple target frameworks. The console application will be built for .NET 5.0 and .NET Core 3.1 (project file CreateNuGet/ConsoleApp/ConsoleApp.csproj):
在创建.NET 控制台应用程序时，您也可以为多个目标框架构建应用程序。与库类似，对于控制台应用程序，您也需要配置多个目标框架。控制台应用程序将针对.NET 5.0 和.NET Core 3.1 构建（项目文件 CreateNuGet/ConsoleApp/ConsoleApp.csproj ）： 

<TargetFrameworks>net5.0; netcoreapp3.1</TargetFrameworks>

An application cannot be built using the .NET Standard 2.0 target framework moniker. Remember, the .NET Standard doesn't contain an implementation of the APIs. If you're using .NET Core 3.1, .NET Standard 2.1 would be okay as well.
不能使用.NET Standard 2.0 目标框架标识符构建应用程序。请记住，.NET Standard 不包含 API 的实现。如果您使用.NET Core 3.1，.NET Standard 2.1 也是可以的。

With the sample application, the same package is needed, but different assemblies from the package need to be selected. This is done automatically based on the project, and the package just needs to be added to the project. The complete project file for the console application is shown here (project file CreateNuGet/DotnetCaller/DotnetCaller.csproj):
对于示例应用程序，需要相同的包，但需要从包中选择不同的程序集。这是根据项目自动完成的，只需将包添加到项目中即可。控制台应用程序的完整项目文件显示在这里（项目文件 CreateNuGet/DotnetCaller/DotnetCaller.csproj ）： 

<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <LangVersion>9.0</LangVersion>
    <Nullable>enable</Nullable>
    <TargetFrameworks>netcoreapp2.0;net47</TargetFrameworks>
  </PropertyGroup>
  <ItemGroup>
    <PackageReference Include="SampleLib" Version="5.0.1" />
  </ItemGroup>
</Project>

With the console application, because the language version is set to 9.0, top-level statements and target-typed new expressions can be used with .NET Core 3.1. The implementation of the console application does not need to be different between the two used frameworks:
在控制台应用程序中，由于语言版本设置为 9.0，可以使用.NET Core 3.1 的顶级语句和目标类型的 new 表达式。控制台应用程序的实现在这两个使用的框架之间不需要有所不同： 

using System;
using SampleLib;
 
Console.WriteLine(Demo.Show());
Book b = new() { Title = "Professional C#", Publisher = "Wrox Press"};
string json = Demo.GetJson(b); 
Console.WriteLine(json);

Building the console application creates multiple binaries that contain references to different libraries. Running the application and setting the --framework to the two options shows two different results. This version:
构建控制台应用程序会创建包含对不同库引用的多个二进制文件。运行应用程序并将 --framework 设置为两个选项会显示两种不同的结果。此版本： 

> dotnet run --framework dotnetcoreapp3.1

results in the following output:
导致以下输出： 

.NET Standard 2.0
{"Title":"Professional C#","Publisher":"Wrox Press"}

Running the .NET 5.0 version, as shown here:
运行 .NET 5.0 版本，如下所示： 

> dotnet run --framework net5.0

results in this output:
产生此输出： 

.NET 5.0
{"Title":"Professional C#","Publisher":"Wrox Press"}

MODULE INITIALIZERS    模块初始化器

In case you need initialization of a library that should be invoked before any class of the library is used, C# 9 has a new feature: module initializers. With a module initializer, the caller doesn't need to invoke any initialization method because it is called automatically before any type of the class is used. A module initializer needs to be a static method without arguments, a void return type, public or internal access modifiers, and the ModuleInitializer attribute applied. This attribute is defined in the namespace System.Runtime.CompilerServices and available only with .NET 5+ (code file UsingLibs/SampleLib/Demo.cs):
如果您需要在使用库的任何类之前初始化库，C# 9 引入了一个新特性：模块初始化器。使用模块初始化器，调用者无需调用任何初始化方法，因为它会在使用类的任何类型之前自动调用。模块初始化器需要是一个无参数的静态方法，返回类型为 void，具有 public 或 internal 访问修饰符，并应用了 ModuleInitializer 属性。此属性定义在 System.Runtime.CompilerServices 命名空间中，仅在.NET 5+可用（代码文件 UsingLibs/SampleLib/Demo.cs ）： 

public class Demo
{
  //…
#if NET50
  [ModuleInitializer]
  internal static void Initializer()
  {
    Console.WriteLine("Module Initializer");
  }
#endif
}

Without using .NET 5, you can create a static constructor instead. However, a static constructor has more runtime overhead and requires that the class is used from the caller. The static constructor will be invoked before the first use of the class, no matter whether static or instance members are invoked. Another option is to define an Initialize method, but this needs to be explicitly invoked by the caller. Module initializers are not called automatically, no matter in which class they are specified.
不使用.NET 5，你可以创建一个静态构造函数。然而，静态构造函数有更高的运行时开销，并且要求调用者使用该类。静态构造函数会在类第一次使用之前被调用，无论调用的是静态成员还是实例成员。另一个选项是定义一个 Initialize 方法，但这需要调用者显式调用。模块初始化器不会自动被调用，无论它们在哪个类中指定。
